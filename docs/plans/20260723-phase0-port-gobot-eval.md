# Phase 0: Port GoBot's static eval exactly to Java (bd nnue-trainer-0dj.1)

## Overview

Cheap decisive experiment for the GoBot-clone epic (`0dj`). Port GoBot's hand-tuned static
evaluation (`backend/search/evaluate.go` in the sibling `../virusgame` repo) **exactly** to
Java — the real function with its literal `EvalParams`, not an NNUE approximation. This removes
the distillation's ~1.5% approximation as a variable and sets up the decisive re-measurement:
run this exact eval with our current negamax search vs GoBot.

**Expected result** (maintainer runs the games, see Post-Completion): still ~0/5 → proves the
GoBot moat is the SEARCH stack, not the eval, and directs Phase 1 (`0dj.2`). If it jumps, the
distillation error mattered. Either way it's a fast, high-information read.

**Scope of THIS plan (ralphex-runnable):** port the eval + prove integer-exact parity against
GoBot via a reference fixture + make it selectable in the search. The GoBot *game* measurement
is maintainer-run (needs live games; ralphex can't do it in a worktree).

## Context (from discovery)

- **Source to port:** `../virusgame/backend/search/evaluate.go` (676 lines). Entry point
  `StaticEval(state game.State, player game.Player) int` → `evaluate` → `evaluateAllWithWorkspace`
  → `analyze` / `spaceRace` / connectivity + base metrics. Hand-tuned weights in
  `defaultEvalParams()` (`Connected:10, Normal:30, Fortified:6, Mobility:1, Captures:1,
  Disconnected:1, BaseExits:180, BaseOpenings:80, BaseAnchors:240, BaseThreat:650,
  ThreatenedLossMult:1, ThreatenedMult:1, SpaceRace:spaceRaceWeight, SealedBasePenalty:5000,
  NeutralUnusedBonus:20, MovesLeftTempo:12, PredatoryCutBase:150, PredatoryCutLossDiv:2`).
  Returns an **integer** score, higher = better for `player`.
- **Parity oracle:** `../virusgame/backend/arena/cmd/staticevalgen/main.go` emits JSONL records
  `{board: [[{owner,kind}]], player, score}` where `score = StaticEval(state, player)`. The board
  encoding matches `import_games.map_board_to_features` / Java `BoardFeatureMapper`. This JSONL is
  the exact-parity test fixture.
- **Java state:** `board/Board.java`, `board/Cell.java`, `board/CellKind.java`;
  `nnue/BoardFeatureMapper.java` already maps our Board to GoBot's cell encoding (precedent for
  the state mapping). Our game is 1v1 (players 1,2) though `evaluate.go` handles up to 4.
- **Search integration point:** `search/SearchEngine.java` currently evaluates leaves via the
  NNUE model; the port must be selectable there.
- ⚠️ **Hidden-state risk:** `StaticEval` may read state beyond the grid (e.g. `MovesLeftTempo`
  uses moves-left; neutral-used flags; current player). `staticevalgen`'s record stores only
  `{board, player, score}`, so the port + any parity test must reproduce whatever extra state
  the oracle used (inspect `staticevalgen` to see what State it builds — likely a fresh turn,
  movesLeft at turn start). If integer parity fails, hidden state is the first suspect.

#### Fixture + hidden-state findings (Task 1 result)

**Fixture:** `src/test/resources/gobot_staticeval_parity.jsonl` — 419 records, one JSONL
line per position: `{"board":[[{owner,kind}...]...],"player","score","movesLeft","neutralUsed"}`.
Sampled every 6th line from a 2509-position self-play run (`-positions 2500 -seed 7`) so it
spans opening→endgame (2..92 owned cells), both movers (p1 232 / p2 187), all `movesLeft`
∈ {1,2,3}, and `neutralUsed` variety ((F,F) 400, (T,F) 13, (F,T) 6). Scores range
−36136..+35882. All 12×12, 2-player, non-terminal (mate-magnitude ≥500M skipped).

**Hidden (non-board) State the score depends on — CRITICAL for the Java port:**
`StaticEval(state, player)` reads more than the grid. Confirmed by reading
`../virusgame/backend/search/evaluate.go`:
- `state.CurrentPlayer()` → stored as `player` (the record's mover). Adds `MovesLeft()*12`
  (MovesLeftTempo) and changes `threatTempo` (current player: `max(1,4-movesLeft)`;
  others: `max(1,movesLeft)`) — so it must equal `player`.
- `state.MovesLeft()` (scalar 0..3) → feeds MovesLeftTempo (weight 12) **and** threatTempo
  for *every* player's threat terms. Varies per position (observed ~uniform over {1,2,3});
  **not board-derivable**.
- `state.NeutralUsed(p)` (per-player bool) → `+20` NeutralUnusedBonus for each active player
  whose neutral is unused; affects both raw[p] and the opponent-subtraction. **Not
  board-derivable.**
- `state.Active(p)` → derivable from the board (an active player has an intact base cell +
  pieces). In these 2-player non-terminal records both players are always active.

Because `movesLeft` and `neutralUsed` are **not** reconstructable from the board yet change
the integer score, the stock `staticevalgen` record (`{board,player,score}` only) is
insufficient for exact parity. The generator was extended to also emit `movesLeft` and
`neutralUsed` (from `state.Snapshot()`), so the fixture is self-contained. The Java port in
Task 2/3 must take `(board, player, movesLeft, neutralUsed[])` and feed those to the eval.

**Regeneration (for the maintainer):** the extra two JSON fields require a one-hunk patch to
`../virusgame/backend/arena/cmd/staticevalgen/main.go` (add `MovesLeft int`/`NeutralUsed
[]bool` to `record`, set from `snap.MovesLeft`/`snap.NeutralUsed` in `toRecord`). Then:
`go build -o /tmp/staticevalgen ./arena/cmd/staticevalgen/ && /tmp/staticevalgen -out
out.jsonl -positions 2500 -seed 7`, sample every 6th line. (This sibling patch is left in the
`../virusgame` working tree, uncommitted — it belongs to that repo, not this one.)

## Technical Details
- Parity means **integer-identical** on the fixture — a hand-tuned eval has no randomness, so
  any diff is a port bug. The fixture is generated from GoBot's own `StaticEval`, so it is the
  ground truth.
- Keep the port a faithful translation (same structure/among-the-same-order operations) to make
  integer parity achievable; don't "improve" or refactor the math.
